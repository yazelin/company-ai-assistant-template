企業內部 AI 助理模板：快速開始

這份文件帶你從 clone 到「打 /ask 拿到答案」走完一遍。每一步都附上「實際指令 → 真實輸出 → 成功的話你會看到」。輸出都是本機真的跑出來貼上的。

前置需求

  • uv（本專案用 uv 管理 Python 環境，會自動準備對應的 Python，pyproject.toml 要求 3.10+）
  • Git
  • 可以使用終端機（會用到 curl；Windows 見下方補充）
  • 之後若要接真實 LLM，再準備 OpenAI 相容的 endpoint 與 API key。第一輪用內建 echo provider，不需要任何金鑰。

環境差異只在「安裝 uv」這一步。裝好之後 uv sync / uv run 在 Ubuntu 與 Windows 完全相同，下面不再每條指令寫兩遍。Windows 的 curl 是 Invoke-WebRequest 的別名，請改用 curl.exe ... 或 Invoke-RestMethod ...；Ubuntu / macOS 用一般 curl。

這個 starter 在做什麼（一句話）

把 sample_docs/ 裡的 .md / .txt 索引進一個 SQLite 檔（assistant.db），然後 POST /ask 時用「關鍵字計次」找出最相關的片段，組成 context 回給你。

注意：assistant.db 沒有放進 repo（已被 .gitignore 忽略）。它是你第一步 ingest 時自己產生的，所以下面 ingest 那步如果沒跑，後面查詢一定是空的。

步驟 0：安裝 uv（一次就好）

Ubuntu / macOS：

curl -LsSf https://astral.sh/uv/install.sh | sh

Windows（PowerShell）：

powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"

裝完重開終端機，uv --version 印得出版本就 OK。

步驟 1：安裝

git clone https://github.com/yazelin/company-ai-assistant-template.git
cd company-ai-assistant-template
uv sync
cp .env.example .env

uv sync 會依 pyproject.toml + uv.lock 自動建立 .venv 並裝好套件（毋須手動 venv/activate），uv sync / uv run 在 Ubuntu 與 Windows 完全相同。.env.example 預設就是 AI_PROVIDER=echo，複製過去即可，先不用改（Windows PowerShell 用 copy .env.example .env）。沒裝 uv 的話 pip install . 也能裝，但本教學以 uv 為主。

步驟 2：把文件索引進 SQLite（這一步會「生出」assistant.db）

uv run python -m app.ingest sample_docs

真實輸出：

ingested sample_docs/overview.md

成功的話你會看到印出 ingested ... 一行，而且資料夾下多出一個 assistant.db 檔（約 12K）。跳過這步的話，後面查詢的 sources 會永遠是空的。

步驟 3：啟動服務

uv run uvicorn app.main:app --reload --port 8000

成功的話終端機會停在 Uvicorn running on http://127.0.0.1:8000 與 Application startup complete.。請另開一個終端機做下面的測試。

步驟 4：健康檢查

curl http://127.0.0.1:8000/health

真實輸出：

{"ok":true}

步驟 5：問一個有命中的問題

curl -X POST http://127.0.0.1:8000/ask \
  -H "Content-Type: application/json" \
  -d '{"question":"What documents does this assistant index?"}'

真實輸出（為了好讀有換行，實際是一行）：

{
  "answer": "Echo answer. Retrieved context:\n[sample_docs/overview.md]\n# Company AI Assistant Template\n\nIndexes internal Markdown/text documents and answers with retrieved context.\n",
  "sources": [
    {
      "score": 2,
      "path": "sample_docs/overview.md",
      "snippet": "# Company AI Assistant Template\n\nIndexes internal Markdown/text documents and answers with retrieved context.\n"
    }
  ]
}

成功的話 sources 陣列裡有東西，path 指到 sample_docs/overview.md。answer 開頭是 Echo answer.，因為現在是 echo provider，它只是把檢索到的 context 原樣回給你，還沒有真的呼叫 LLM。這是預期行為。

步驟 6：問一個沒命中的問題

curl -X POST http://127.0.0.1:8000/ask \
  -H "Content-Type: application/json" \
  -d '{"question":"zzzzzz nonexistent topic"}'

真實輸出：

{"answer":"Echo answer. Retrieved context:\nNo matching docs.","sources":[]}

成功的話 sources 是空陣列 []，answer 是 No matching docs.。代表檢索正常運作，只是這個問題在你目前的文件裡找不到關鍵字。

看看錯誤長什麼樣

不存在的路徑（404）：curl http://127.0.0.1:8000/asdf

{"detail":"Not Found"}

打 /ask 但忘了帶 question 欄位（422 驗證錯誤）：

{"detail":[{"type":"missing","loc":["body","question"],"msg":"Field required","input":{}}]}

422 不是壞掉，是 request body 少了必填欄位。

整體 OK 的判斷標準

/health 回 {"ok":true}、有命中的問題回得到非空 sources、沒命中的問題回 "sources":[]，就代表這個 starter 在你機器上完整跑通了。

企業內部 AI 助理模板：架構說明

核心檔案

  • app/ingest.py：讀取 .md / .txt 並寫入 SQLite documents table
  • app/search.py：最小 keyword retrieval baseline
  • app/ai.py：把檢索片段組成 context，再交給 echo 或 HTTP LLM
  • app/main.py：提供 /ask API
  • sample_docs/：示範用內部文件

資料流

  1. client 發出 POST /ask 請求。
  2. FastAPI 接收並用 Pydantic 驗證 question 欄位。
  3. search_docs 對 SQLite 做關鍵字計次檢索。
  4. ai.answer 把片段組成 context，交給 echo 或 HTTP LLM。
  5. 回傳 answer 與 sources。

設計原則

  • 先讓流程可跑，再做漂亮抽象。
  • token 與 secrets 全部放在環境變數。
  • 每一層保持可以替換：入口、AI provider、資料來源、部署方式。
  • 範例程式刻意保持小，方便你看懂後改成自己的版本。

企業內部 AI 助理模板：帶你走一遍

quickstart 讓服務跑起來。這份文件帶你做兩件真正有價值的事：把「你自己的文件」加進去並證明它被檢索到（含真實 before / after），以及從 echo 換成真實 HTTP LLM。下面的輸出都是本機真的跑出來貼上的。

第一部分：加一份自己的文件，看它被檢索到

Before：目前只有 overview.md，先問一個沒有的主題：

curl -X POST http://127.0.0.1:8000/ask \
  -H "Content-Type: application/json" \
  -d '{"question":"refund window thirty days"}'

真實輸出（sources 是空的）：

{"answer":"Echo answer. Retrieved context:\nNo matching docs.","sources":[]}

動作：在 sample_docs/ 新增 refund-policy.md：

# Refund Policy

Customers may request a refund within a thirty day window after purchase.
Approved refunds are returned to the original payment method.

重新 ingest（改完文件一定要重跑，否則 SQLite 裡還是舊內容）：

uv run python -m app.ingest sample_docs

真實輸出（多了一行新檔案）：

ingested sample_docs/overview.md
ingested sample_docs/refund-policy.md

After：問同一個問題，真實輸出：

{
  "answer": "Echo answer. Retrieved context:\n[sample_docs/refund-policy.md]\n# Refund Policy\n...",
  "sources": [
    {
      "score": 5,
      "path": "sample_docs/refund-policy.md",
      "snippet": "# Refund Policy\n\nCustomers may request a refund within a thirty day window after purchase.\nApproved refunds are returned to the original payment method.\n"
    }
  ]
}

同一個問題，sources 從 [] 變成命中 sample_docs/refund-policy.md，score 是 5。這就是「加文件 → 重新 ingest → 被檢索到」的完整證明。

提醒：這個 starter 的檢索是「關鍵字 / 子字串計次」（看 app/search.py），不是語意檢索。中文「退費」不會自動對應英文 refund，同義詞也命中不到。這是個誠實的 keyword retrieval baseline。

第二部分：從 echo 換成真實 HTTP LLM

把 .env 的 AI_PROVIDER 從 echo 改成 http，並填入 HTTP_LLM_API_KEY：

AI_PROVIDER=http
HTTP_LLM_ENDPOINT=https://api.openai.com/v1/chat/completions
HTTP_LLM_API_KEY=sk-你的金鑰
MODEL_NAME=gpt-4o-mini

app/ai.py 的邏輯：只要 AI_PROVIDER 不是 echo，就用 httpx 對 HTTP_LLM_ENDPOINT 發一個 OpenAI 相容的 chat completions 請求，把檢索片段塞進 prompt，回傳 choices[0].message.content。endpoint 不限 OpenAI，任何 OpenAI 相容介面都行（vLLM、Ollama 的相容層、Groq 等）。

這一步需要你自己的金鑰才能真的拿到模型回應，所以這裡不貼造假的 LLM 回應。設好你的 HTTP_LLM_API_KEY 後，重啟 uvicorn 再打同一個 /ask，你會看到 answer 不再是 "Echo answer..."，而是模型根據 sources 寫出的自然語言回答；sources 欄位仍照舊回傳。金鑰沒設好或無效時 /ask 會回 500，錯誤訊息見踩雷清單。

動手練習：在 /ask 回應多加一個 source_count 欄位

目標：讓 /ask 除了 answer / sources，再回一個「命中幾份文件」的數字。這個練習我實際做過、驗證有效，下面是真實結果。

改 app/main.py，在回傳 dict 多一個 source_count：

return {"answer":await answer(req.question,docs),"sources":docs,"source_count":len(docs)}

驗證真實輸出（有命中）：

{"answer":"Echo answer. Retrieved context:\n[sample_docs/refund-policy.md]\n...","sources":[{"score":5,"path":"sample_docs/refund-policy.md","snippet":"..."}],"source_count":1}

沒命中：

{"answer":"Echo answer. Retrieved context:\nNo matching docs.","sources":[],"source_count":0}

成功的話回應尾巴多了 "source_count":1（命中一份）和 "source_count":0（沒命中）。一個三個字的修改就讓 API 多回一個有用欄位，這就是這個 starter 刻意保持小的好處。

升級路徑

  • 檢索：keyword 計次 → chunking → embedding 向量檢索 → hybrid。這是「baseline → RAG」真正的那段。
  • 中文 / 同義詞：keyword baseline 不處理，要在 ingest 做正規化或直接上 embedding。
  • 權限：目前任何人打 /ask 都能讀全部文件，正式版要加登入與依部門分群。
  • 資料治理：insert or replace 不處理刪除與版本，正式版要補。

企業內部 AI 助理模板：部署筆記

部署前檢查

  • 本機流程已經跑通。
  • .env 沒有被 commit。
  • README 的啟動指令與實際程式一致。
  • /health 可用。
  • 外部 LLM 的 HTTP_LLM_API_KEY 已放在部署平台 secrets，而不是寫死在程式或 commit。
  • 部署目標上有跑過一次 ingest（assistant.db 是執行時產生的，不在 repo 裡）。

啟動指令

部署機先 uv sync（依 uv.lock 還原相同套件版本），再用 uv run 啟動。

uv sync
uv run uvicorn app.main:app --host 0.0.0.0 --port 8000

常見部署選項

  • Render / Railway / Fly.io：適合快速 demo。
  • VPS + Docker / systemd：適合長期自管。
  • 公司內網主機：適合企業內部工具，但要處理網路與權限。
  • NAS / edge gateway：適合工業或內部自動化場景。

部署後驗證

curl http://127.0.0.1:8000/health

接著打一個有命中的 /ask 確認 sources 非空，不要只看服務有沒有啟動。

實務提醒

部署不是最後一步。正式使用前至少要補：log、錯誤告警、權限控管、備份策略、secret rotation，以及基本監控。

企業內部 AI 助理模板：常見問題與踩雷清單

下面每一條都是這個 repo 真的會踩到的坑，錯誤訊息都是本機實際跑出來貼上的。

坑 1：忘了先 ingest，查詢永遠是空的

assistant.db 不在 repo 裡，是 ingest 時才產生的。沒跑 ingest 就查，sources 一定是空陣列，而且不會報錯（db.py 會自動建空表）。對一個從沒 ingest 過的 DB 檢索，真實結果是 []。修法：先跑 uv run python -m app.ingest sample_docs。

坑 2：改了文件卻沒重新 ingest

/ask 讀的是 SQLite 不是直接讀檔。改了 sample_docs 沒重新 ingest，查到的還是舊資料。每次動 sample_docs 後都重跑 ingest（insert or replace，重跑安全）。

坑 3：DATABASE_URL 指到別的路徑

db.py 讀 DATABASE_URL（預設 assistant.db，相對於執行目錄）。ingest 和 uvicorn 在不同目錄或用了不同 .env，會操作到不同的 .db，於是「明明 ingest 成功，查還是空的」。固定在 repo 根目錄執行所有指令。

坑 4：AI_PROVIDER 設成 http 但金鑰沒填

送出 Authorization: Bearer（後面空白），httpx 直接擋下來，真實錯誤：

LocalProtocolError: Illegal header value b'Bearer '

/ask 回 500。修法：填上有效金鑰，或先改回 AI_PROVIDER=echo 排除 LLM 問題。

坑 5：金鑰 / endpoint 設錯（401 等）

金鑰無效或 endpoint 錯，會在 raise_for_status 丟出帶狀態碼的錯誤，真實錯誤：

HTTPStatusError: Client error '401 Unauthorized' for url 'https://api.openai.com/v1/chat/completions'

401 = 金鑰錯或過期；404 / 連線錯誤 = endpoint 填錯。

坑 6：request body 少欄位（422）

{"detail":[{"type":"missing","loc":["body","question"],"msg":"Field required","input":{}}]}

這是正常的輸入驗證，帶上 question 即可。

坑 7：以為 echo 就是 AI

echo provider 不呼叫任何模型，只把檢索 context 原樣回傳（answer 開頭固定是 Echo answer.）。要真的讓模型回答，把 provider 換成 http。

坑 8：忘了先 uv sync，或沒在指令前加 uv run

本專案改用 uv，相依套件裝在 uv sync 自動建立的 .venv 裡。沒先 uv sync 就跑，或直接打 uvicorn ... / python -m app.ingest ...（少了 uv run），會用到系統 Python，常見錯誤是 ModuleNotFoundError: No module named 'fastapi' 或 uvicorn: command not found。修法：在 repo 根目錄先 uv sync 一次，之後所有指令都加 uv run 前綴。

坑 9：uv: command not found（uv 沒裝或沒重開終端機）

還沒裝 uv，或裝完沒重開終端機（installer 會把 uv 加進 PATH，需要新的 shell 才生效）。修法：Ubuntu / macOS 跑 curl -LsSf https://astral.sh/uv/install.sh | sh；Windows（PowerShell）跑 powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"，裝完重開終端機再 uv --version 確認。

企業內部 AI 助理模板：改成你的使用場景

可延伸方向

  • 改成部門知識庫：依文件路徑、部門、權限分群。
  • 升級檢索：加入 chunking、embedding、hybrid search。
  • 加登入：用公司 SSO 或 API key 控制 /ask。
  • 加評估集：收集常見問題，定期測回答品質。

改造原則

  • 一次只改一個層次：先改資料來源，再改 UI，再改部署。
  • 先保留原本可跑的範例，另開 branch 做實驗。
  • 每加一個外部服務，就加一個健康檢查或 smoke test。
  • 先做 PoC，再決定要不要產品化。
